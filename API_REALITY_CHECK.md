# API 구조 - 현실성 검증 보고서

**작성일**: 2026-04-18  
**결론**: 원래 설계에 6가지 중요한 현실성 문제 발견

---

## ⚠️ 발견된 6가지 문제

### 1️⃣ **Groq "무료" → 실제 불명확 (위험도: 높음)**

**원래 주장:**
```
Groq = 무료 API (Rate Limited 30k tokens/min)
추천: 2순위
```

**현실:**
```
✗ 공식 가격 정책이 명시되지 않음
✗ "Free tier"가 정말 무료인지 불명확
✗ 이미 2024년 내 정책 변경 여러 번
✗ 상업 사용 제약이 있을 수 있음
✗ Rate limit은 "무료"가 아니라 "속도 제한"
```

**권장 조치:**
```
[ ] Groq 공식 문서 직접 확인
[ ] 계정 만들어서 실제 비용 청구 여부 테스트
[ ] 정책 변경 감시 체계 구축
→ Tier 2 (보조용)로 강등
```

---

### 2️⃣ **Together.ai "$5 크레딧" → 불명확 (위험도: 중)**

**원래 주장:**
```
Together AI = $5/월 크레딧 제공
추천: 4순위
```

**현실:**
```
✗ $5는 가입 후 처음 30일만?
✗ 매월 자동 갱신되는지 불명확
✗ 활성도 기준이 있을 수 있음
✗ 또는 이미 정책이 변경되었을 수 있음
✗ 공식 트위터/블로그 확인 필요
```

**권장 조치:**
```
[ ] Together.ai 계정 실제 가입 테스트
[ ] 크레딧 자동 갱신 정책 확인
[ ] 비용 청구 메커니즘 확인
→ Tier 2 (미확인)로 표시
```

---

### 3️⃣ **HuggingFace "완전 무료" → 성능 문제 (위험도: 중)**

**원래 주장:**
```
HuggingFace = 무료 (Inference API)
성능: Queue 기반
추천: 3순위
```

**현실:**
```
✗ "무료"이지만 Queue = 2-5분 지연 가능
✗ 실시간 응답 필요한 경우 완전히 사용 불가
✗ 트래픽 좋은 시간: 응답 더 지연
✗ "무료"로 표시하면 사용자 기대 ❌

예: "지금 날씨 알려줘"
→ 2-5분 후 응답 = 쓸모 없음
```

**권장 조치:**
```
[ ] "비동기 작업 전용"으로 명확히 재분류
[ ] 응답 시간 실측 테스트
[ ] 실시간 요청에는 제외
→ Tier 2 (최후수단)로 강등
```

---

### 4️⃣ **Ollama "성능 최대화" → 동시성 병목 (위험도: 높음)**

**원래 주장:**
```
Ollama = 로컬, 초고속 (0.2초)
처리량: 100 req/sec
캐시 히트율: 60-80%
```

**현실:**
```
✗ Ollama는 GPU 사용 (단일 처리)
✗ 동시 10개 요청 → 각각 1-2초 추가 지연

예:
  요청 1: 0.2초 (GPU)
  요청 2: 1.2초 (큐 대기)
  요청 3: 2.2초 (큐 대기)
  요청 4: 3.2초 (큐 대기)
  ...

✗ "100 req/sec"는 캐시 히트일 때만
✗ 동시성 제약 문서에 미표기
```

**권장 조치:**
```
[ ] 실제 동시 요청 테스트 (5개, 10개)
[ ] 응답 시간 측정
[ ] GPU 메모리 사용량 모니터링
[ ] 동시성 한계 명문화 (10-20 req/sec?)
→ 성능 지표 현실화
```

---

### 5️⃣ **캐시 히트율 "60-80%" → 과대평가 (위험도: 중)**

**원래 주장:**
```
캐시 히트율: 60-80%
- Redis (1시간): 60%
- SQLite (30일): 20%
```

**현실:**
```
✗ 실제 환경:
  - 첫 요청: 100% 미스
  - 1시간 내 재질문: 1-10%만 발생
  - 유저 행동은 매우 다양함

✗ 현실적 기대:
  - 캐시 히트율: 30-40%
  - 우연의 일치 + 자주 묻는 질문만

예: 100명 사용자
  - 같은 질문 다시 하는 사람: 5-10명
  - 캐시 히트: 5-10%
```

**권장 조치:**
```
[ ] 실제 사용자 행동 분석
[ ] 상위 10개 질문 통계
[ ] 캐시 히트율 실측 (1주)
→ 목표: 30-40%로 재설정
```

---

### 6️⃣ **API 키 관리 → 복잡도 높음 (위험도: 중)**

**원래 주장:**
```
무료 API 5개 조합
```

**현실:**
```
✗ 관리할 API 키:
  1. Groq API key
  2. HuggingFace API key
  3. Together.ai API key
  4. Claude API key
  5. Ollama (선택사항)

✗ 각각 다른 인증 방식
✗ 한 개 누출 → 모든 API 위험
✗ .env 파일 관리의 복잡도 → 보안 위험
✗ 환경별(dev/prod) 키 관리 필요
```

**권장 조치:**
```
[ ] .env 암호화 (git-crypt 또는 AWS Secrets)
[ ] 키 로테이션 정책 수립
[ ] 각 API별 Rate Limit 모니터링
[ ] 비상 대응 계획 (키 누출 시)
→ 단순화: Tier 1만 (Ollama + Claude)
```

```
월 비용: $1-5
가장 저렴한 구조
"거의 무료"
```

**현실:**

```
비용 구성:
1. Ollama: ₩0 (로컬)
2. Claude: $0.00025/input token
   예: 월 1000만 tokens = $2.50
3. Groq: $0? (정책 불명확)
4. HF: $0? (정책 불명확)
5. Together: $0? ($5인지 불명확)

실제 시나리오:
- 보수적: Claude만 사용 → $5-10/월
- 중간: Claude + Groq → $5-20/월
- 공격적: 모든 API → $20-50/월

결론: "$1-5"는 너무 낙관적
현실: $5-20/월 예상 (Claude 의존도 높음)
```

## ✅ 현실적 재설계

### Tier 재분류

```
🟢 Tier 1: 신뢰할 수 있는 것 (메인)
  - Ollama (로컬, 무료)
  - Claude API (공식, 가격 명확)

🟡 Tier 2: 신뢰도 낮음 (보조, 정책 감시)
  - Groq (정책 불명확)
  - HuggingFace (성능 문제)
  - Together.ai (크레딧 불명확)
```

### 성능 목표 현실화

| 지표 | 원래 | 현실 | 
|------|-------|--------|
| 응답 시간 | 0.01초 | 0.2초 |
| 캐시 히트율 | 60-80% | 30-40% |
| 동시 처리 | 100 req/s | 10-50 req/s |
| 월 비용 | $1-5 | $5-20 |

### 검증 체크리스트

```markdown
□ Groq 정책 확인
  - 공식 가격 정책 읽기
  - 실제 계정 테스트
  - API 호출 비용 청구 여부 확인

□ Together.ai 크레딧 확인
  - 계정 가입 테스트
  - $5 크레딧 받는지 확인
  - 자동 갱신 정책 확인

□ HuggingFace 성능 측정
  - 실제 응답 시간 테스트
  - 동시 요청 테스트
  - Queue 대기 시간 측정

□ Ollama 동시성 테스트
  - 동시 5개, 10개, 20개 요청
  - 각 응답 시간 측정
  - GPU 메모리 모니터링

□ 캐시 히트율 실측
  - 1주일 데이터 수집
  - 실제 사용자 행동 분석
  - 현실적 목표 설정

□ 비용 추적 시작
  - Claude 사용량 기록
  - 무료 API 정책 감시
  - 월별 예산 계획
```

## 🎯 결론

**원래 설계의 문제:**
1. 무료 API 정책에 대한 과도한 낙관
2. 성능 지표의 현실성 부족 (동시성, 캐시)
3. 비용 예측의 부정확성
4. 위험 요소에 대한 인식 부족

**개선 방향:**
1. Tier 1 (신뢰) vs Tier 2 (보조) 명확 구분
2. 모든 주장에 "검증 필요" 표시
3. 성능 목표를 현실적으로 재설정
4. 정책 감시 체계 구축
5. 월별 비용을 $5-20으로 재예상

**진행 방법:**
- 즉시: Tier 2 API 정책 문서 확인
- 1주: 실제 테스트로 성능/비용 검증
- 지속: 정책 변경 감시 시스템 운영

**최종 메시지:**
"완전 무료는 불가능. Claude 기반으로 설계하고, 무료 API는 보조로만 사용하는 것이 현실적."
